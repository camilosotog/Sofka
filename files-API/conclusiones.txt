PROBLEMAS IDENTIFICADOS

1. Código de respuesta en creación
La API retorna 200 OK al crear una mascota (POST /pet). El estándar REST indica que debería retornar 201 Created para diferenciar una creación exitosa de una simple operación exitosa.

2. Comportamiento upsert en PUT
El endpoint PUT /pet no valida si el recurso existe previamente. Si se envía un ID inexistente, la API crea un nuevo recurso en lugar de retornar 404 Not Found. Esto puede generar datos duplicados o inconsistentes.

3. Sin autenticación obligatoria
Los endpoints permiten crear, modificar y consultar mascotas sin autenticación. Aunque la API provee un mecanismo de API Key, no lo exige, lo que en un entorno productivo sería una vulnerabilidad de seguridad.

4. Sin validación de datos
La API no valida campos obligatorios. Es posible crear una mascota sin nombre o sin photoUrls, lo cual podría generar datos incompletos en la base de datos.

- Framework: Playwright Test con TypeScript
- Ejecución: test.describe.serial() para flujo secuencial (TC-01 → TC-02 → TC-03 → TC-04)
- Datos: Externalizados en CSV (tests/data/csv/petstore-tests.csv), leídos con csv-reader.ts
- Variables compartidas: petId y petName se pasan entre tests como variables de módulo
- Reporte: Cada test tiene 4 steps (Entradas, Salidas capturadas, Variables/fixtures, Assertions) con datos adjuntos en JSON via test.info().attach()
- Reporters: HTML, List y JSON

CONCLUSIONES

Las pruebas de API contra PetStore Swagger confirmaron que el flujo CRUD completo (Crear → Consultar → Actualizar → Buscar) funciona correctamente. Los 4 endpoints responden con status 200 y retornan los datos esperados.

Los tests se diseñaron como un flujo serial donde cada caso depende del anterior: TC-01 crea la mascota y extrae el petId, TC-02 la consulta por ese ID, TC-03 la actualiza, y TC-04 verifica que aparezca en la búsqueda por status. Esto simula un escenario real de uso de la API.

Los datos de prueba están externalizados en un archivo CSV, evitando valores quemados en el código y facilitando la modificación de datos sin alterar los tests.

Los hallazgos principales son: el uso de 200 en lugar de 201 para creación, el comportamiento upsert del PUT que no valida existencia, y la falta de autenticación obligatoria. Estos no son bugs funcionales, sino observaciones de diseño que deberían considerarse en una API productiva.
